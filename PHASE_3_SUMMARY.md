# Phase 3 Summary: Actions + Manual Score Entry

## What Was Implemented

Phase 3 added scoring actions (the rules that define what earns or deducts points) and a manual score entry interface. The admin defines actions, then uses the Score Entry tab to award points by entering a participant code and an action code. All transactions are recorded in an immutable audit log.

## Database Changes

### Migration: `003_create_actions_transactions.sql`

**Table created: `actions`**

| Column | Type | Constraints |
|---|---|---|
| `id` | UUID | PRIMARY KEY, auto-generated |
| `event_id` | UUID | NOT NULL, FK to `events(id)` ON DELETE CASCADE |
| `code` | TEXT | NOT NULL (auto-generated as `A-XXXX` — added in Phase 4 migration) |
| `name` | TEXT | NOT NULL |
| `points` | INTEGER | NOT NULL (can be negative for deductions) |
| `description` | TEXT | Nullable |
| `is_active` | BOOLEAN | NOT NULL, DEFAULT `true` |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT `now()` |
| `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT `now()` |

**Table created: `point_transactions`**

| Column | Type | Constraints |
|---|---|---|
| `id` | UUID | PRIMARY KEY, auto-generated |
| `event_id` | UUID | NOT NULL, FK to `events(id)` ON DELETE CASCADE |
| `participant_id` | UUID | NOT NULL, FK to `participants(id)` ON DELETE CASCADE |
| `action_id` | UUID | NOT NULL, FK to `actions(id)` ON DELETE CASCADE |
| `points` | INTEGER | NOT NULL (snapshot of action's points at transaction time) |
| `created_by` | UUID | NOT NULL, FK to `auth.users(id)` |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT `now()` |

**Indexes:**
- `idx_actions_event_code` — UNIQUE on `(event_id, code)`. One action code per event.
- `idx_actions_event_id` — Regular on `event_id`.
- `idx_point_transactions_event_created` — On `(event_id, created_at DESC)`. Supports recent transaction queries.

**Triggers:**
- `actions_updated_at` — Auto-updates `updated_at` on actions.
- `point_transactions_no_update` — BEFORE UPDATE. Calls `prevent_point_transaction_update()`. Raises exception: "Point transactions cannot be modified".
- `point_transactions_no_delete` — BEFORE DELETE. Calls `prevent_point_transaction_delete()`. Raises exception: "Point transactions cannot be deleted".
- `point_transactions_same_event` — BEFORE INSERT. Calls `check_transaction_same_event()`. Validates that participant and action belong to the same event as the transaction.

**Functions:**
- `prevent_point_transaction_update()` — Always raises exception. Makes transactions immutable.
- `prevent_point_transaction_delete()` — Always raises exception. Makes transactions immutable.
- `check_transaction_same_event()` — Validates that `participant.event_id` and `action.event_id` both match `NEW.event_id`.

**RLS Policies:**

On `actions`:
- SELECT: Admin can view own event actions
- INSERT: Admin can create actions for own event
- UPDATE: Admin can update own event actions
- No DELETE policy (action deletion is denied by default)

On `point_transactions`:
- SELECT: Admin can view own event transactions
- INSERT: Admin can create transactions for own event
- No UPDATE or DELETE policies (additionally blocked by triggers)

## UI Pages

No new pages. All functionality is within the Dashboard as the "Actions" and "Score" tabs.

## Components

### Action Components

**`ActionList`** (`src/components/actions/ActionList.tsx`)
- Loads all actions for the event, sorted by `created_at` ascending
- Query: `actions.select('*').eq('event_id', eventId)`
- Supports: create, edit, toggle active/inactive
- Reports count to dashboard via `onCountChange` callback

**`ActionForm`** (`src/components/actions/ActionForm.tsx`)
- Modal form for adding/editing actions
- Create mode: fields for name, points, description. Code is auto-generated by the database.
- Edit mode: shows action code as read-only text, plus editable name, points, description, and active checkbox.
- Validation: name required, points must be a valid integer.
- Insert: sends `{ event_id, name, points, description }` — `code` is set by a BEFORE INSERT trigger (Phase 4 migration).
- Update: sends `{ name, points, description, is_active }` — code is immutable.

**`ActionRow`** (`src/components/actions/ActionRow.tsx`)
- Displays: action code (monospace badge), name, description (truncated), points badge (green for >= 0, red for < 0)
- Points show `+` prefix for non-negative values
- Inactive actions render at 60% opacity
- Action buttons: Edit, Activate/Deactivate toggle

### Scoring Components

**`ScoreEntry`** (`src/components/scoring/ScoreEntry.tsx`)
- Two-field form: Participant Code and Action Code
- Submission flow:
  1. Looks up participant by `external_id` in the event
  2. Looks up action by `code` in the event
  3. Validates participant exists
  4. Validates action exists
  5. Validates action is active
  6. Inserts `point_transaction` with `{ event_id, participant_id, action_id, points, created_by }`
- On success: shows green message "Awarded +X points to [Name] for [Action]", clears form, refocuses participant input
- On error: shows red error message
- Below the form: displays 20 most recent transactions with full details

**`TransactionRow`** (`src/components/scoring/TransactionRow.tsx`)
- Displays: participant name + code, action name + code, timestamp, points badge (green/red)
- Timestamp formatted via `toLocaleString()`

## APIs / Supabase Queries

| Operation | Table | Method | Details |
|---|---|---|---|
| List actions | `actions` | SELECT | All for event, ordered by `created_at` ASC |
| Create action | `actions` | INSERT | `{ event_id, name, points, description }` |
| Update action | `actions` | UPDATE | `{ name, points, description, is_active }` by id |
| Toggle active | `actions` | UPDATE | `{ is_active: !current }` by id |
| Find participant by code | `participants` | SELECT | `.eq('event_id', ...).eq('external_id', code).maybeSingle()` |
| Find action by code | `actions` | SELECT | `.eq('event_id', ...).eq('code', code).maybeSingle()` |
| Create transaction | `point_transactions` | INSERT | `{ event_id, participant_id, action_id, points, created_by }` |
| List recent transactions | `point_transactions` | SELECT | Joined with `participants(name, external_id)` and `actions(name, code)`, ordered by `created_at` DESC, limit 20 |

## Business Rules

1. **Unique action codes per event.** Enforced by UNIQUE index on `(event_id, code)`.
2. **Action code is read-only after creation.** The UI does not allow editing the auto-generated code.
3. **Negative points are allowed.** Actions can have negative point values for deductions.
4. **Points are snapshot at transaction time.** The `points` column on `point_transactions` stores the action's point value at the time of the transaction. If the action's points are later changed, existing transactions are unaffected.
5. **Transactions are immutable.** Triggers prevent UPDATE and DELETE on `point_transactions`. Once a score is recorded, it cannot be modified or removed.
6. **Active actions only.** The Score Entry form rejects transactions for inactive actions with the message "Action [code] is inactive".
7. **Same-event validation.** The `check_transaction_same_event()` trigger ensures the participant, action, and transaction all reference the same event.
8. **No action deletion.** No DELETE RLS policy exists on `actions`. Actions can only be deactivated, not deleted.
9. **`created_by` records the admin.** Every transaction records which admin created it via `auth.users(id)` FK.

## What Is Not Implemented

- No bulk score entry.
- No undo/reverse transaction.
- No transaction filtering or search beyond the most recent 20.
- No action deletion (by design — deactivation is the intended workflow).
- No validation that the same participant+action combination hasn't been scored before (duplicate transactions are allowed).
