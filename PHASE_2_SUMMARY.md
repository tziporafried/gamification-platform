# Phase 2 Summary: Groups + Participants

## What Was Implemented

Phase 2 added the ability to manage participants and groups within an event. Participants are registered with a name, assigned a system-generated code, and can be placed into one or more color-coded groups. A many-to-many relationship between participants and groups is supported via a junction table.

## Database Changes

### Migration: `002_create_groups_participants.sql`

**Table created: `groups`**

| Column | Type | Constraints |
|---|---|---|
| `id` | UUID | PRIMARY KEY, auto-generated |
| `event_id` | UUID | NOT NULL, FK to `events(id)` ON DELETE CASCADE |
| `name` | TEXT | NOT NULL |
| `color` | TEXT | NOT NULL, DEFAULT `'#6366f1'` |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT `now()` |
| `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT `now()` |

CHECK constraint: `groups_color_hex` — `color ~* '^#[0-9a-f]{6}$'` (must be a valid 6-digit hex color).

**Table created: `participants`**

| Column | Type | Constraints |
|---|---|---|
| `id` | UUID | PRIMARY KEY, auto-generated |
| `event_id` | UUID | NOT NULL, FK to `events(id)` ON DELETE CASCADE |
| `external_id` | TEXT | NOT NULL (auto-generated as `P-XXXX` — added in Phase 4 migration) |
| `name` | TEXT | NOT NULL |
| `email` | TEXT | Nullable (removed in Phase 4 migration) |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT `now()` |
| `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT `now()` |

**Table created: `participant_groups` (junction)**

| Column | Type | Constraints |
|---|---|---|
| `participant_id` | UUID | NOT NULL, FK to `participants(id)` ON DELETE CASCADE |
| `group_id` | UUID | NOT NULL, FK to `groups(id)` ON DELETE CASCADE |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT `now()` |

PRIMARY KEY: `(participant_id, group_id)`

**Indexes:**
- `idx_groups_event_name` — UNIQUE on `(event_id, name)`. One group name per event.
- `idx_groups_event_id` — Regular on `event_id`.
- `idx_participants_event_external_id` — UNIQUE on `(event_id, external_id)`. One participant code per event.
- `idx_participants_event_id` — Regular on `event_id`.
- `idx_participant_groups_group_id` — Regular on `group_id`.

**Triggers:**
- `groups_updated_at` — Auto-updates `updated_at` on groups.
- `participants_updated_at` — Auto-updates `updated_at` on participants.

**Functions:**
- `check_participant_group_same_event()` — BEFORE INSERT trigger on `participant_groups`. Validates that the participant and group belong to the same event. Raises exception if they do not.

**RLS Policies:**

On `groups`:
- SELECT: Admin can view own event groups
- INSERT: Admin can create groups for own event
- UPDATE: Admin can update own event groups
- DELETE: Admin can delete own event groups

On `participants`:
- SELECT: Admin can view own event participants
- INSERT: Admin can create participants for own event
- UPDATE: Admin can update own event participants
- DELETE: Admin can delete own event participants

On `participant_groups`:
- SELECT: Admin can view own event assignments (checks via participant's event)
- INSERT: Admin can create assignments for own event (checks via participant's event)
- DELETE: Admin can delete own event assignments (checks via participant's event)
- No UPDATE policy (assignments are created or deleted, not modified)

## UI Pages

No new pages were added. All functionality is within the existing Dashboard page as tabs.

## Components

### Participant Components

**`ParticipantList`** (`src/components/participants/ParticipantList.tsx`)
- Loads all participants for the event with nested group relationships
- Query: `participants.select('*, participant_groups(group_id, groups(*))').eq('event_id', eventId)`
- Maps the nested join result to `ParticipantWithGroups[]`
- Supports: create, edit, delete, manage group assignments
- Delete shows a confirmation modal warning that group assignments will also be removed

**`ParticipantForm`** (`src/components/participants/ParticipantForm.tsx`)
- Modal form for adding/editing participants
- Create mode: single "Name" field. The participant code is auto-generated by the database.
- Edit mode: shows the participant code as read-only text, plus editable "Name" field.
- Insert: sends `{ name, event_id }` — `external_id` is set by a BEFORE INSERT trigger (Phase 4 migration).
- Update: sends `{ name }` only — code is immutable after creation.

**`ParticipantRow`** (`src/components/participants/ParticipantRow.tsx`)
- Displays: participant code (monospace badge), name, group badges (colored)
- Action buttons: Groups, Edit, Delete

### Group Components

**`GroupList`** (`src/components/groups/GroupList.tsx`)
- Loads all groups with member counts via aggregation
- Query: `groups.select('*, participant_groups(count)').eq('event_id', eventId)`
- Supports: create, edit, delete
- Delete shows a confirmation modal

**`GroupForm`** (`src/components/groups/GroupForm.tsx`)
- Modal form for adding/editing groups
- Fields: name, color (via `ColorPicker`)
- Detects duplicate group name within event (error code `23505`)

**`GroupCard`** (`src/components/groups/GroupCard.tsx`)
- Card display with colored top bar, avatar with first letter, group name, member count
- Action buttons: Edit, Delete

**`GroupAssignment`** (`src/components/groups/GroupAssignment.tsx`)
- Modal for assigning a participant to groups
- Loads all event groups and current participant's group assignments
- Checkbox toggle to add/remove from each group
- Optimistic UI with rollback on error
- Insert: `participant_groups.insert({ participant_id, group_id })`
- Delete: `participant_groups.delete().eq('participant_id', ...).eq('group_id', ...)`

## APIs / Supabase Queries

| Operation | Table | Method | Details |
|---|---|---|---|
| List participants with groups | `participants` | SELECT | Nested join through `participant_groups` to `groups` |
| Create participant | `participants` | INSERT | `{ name, event_id }` |
| Update participant | `participants` | UPDATE | `{ name }` by id |
| Delete participant | `participants` | DELETE | By id. Cascades to `participant_groups`. |
| List groups with counts | `groups` | SELECT | Aggregates `participant_groups(count)` |
| Create group | `groups` | INSERT | `{ name, color, event_id }` |
| Update group | `groups` | UPDATE | `{ name, color }` by id |
| Delete group | `groups` | DELETE | By id. Cascades `participant_groups`. |
| List participant's groups | `participant_groups` | SELECT | Filtered by `participant_id` |
| Assign to group | `participant_groups` | INSERT | `{ participant_id, group_id }` |
| Remove from group | `participant_groups` | DELETE | By `participant_id` + `group_id` |

## Business Rules

1. **Unique group names per event.** Enforced by UNIQUE index on `(event_id, name)`.
2. **Unique participant codes per event.** Enforced by UNIQUE index on `(event_id, external_id)`.
3. **Many-to-many participant-group relationship.** A participant can belong to multiple groups. A group can contain multiple participants.
4. **Cross-event guard.** The `check_participant_group_same_event()` trigger prevents assigning a participant to a group from a different event.
5. **Cascade deletes.** Deleting a participant removes all its group assignments. Deleting a group removes all its participant assignments.
6. **Valid hex color required.** The `groups_color_hex` CHECK constraint ensures group colors are valid 6-digit hex values.
7. **Participant code is read-only after creation.** The UI does not allow editing the auto-generated code. The code is shown as non-editable text in edit mode.

## What Is Not Implemented

- No bulk participant import (CSV, etc.).
- No participant search or filtering.
- No group reordering.
- No limit on the number of groups a participant can belong to.
